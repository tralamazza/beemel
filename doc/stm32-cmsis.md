# STM32 + CMSIS workflow

End-to-end recipe for using STM's per-family `cmsis-device-fX` GitHub repos with
bml: generate a `.target` file, generate `peripheral` declarations, and (optionally)
call the vendor `SystemInit()` for clock setup.

## 1. Overview

STM publishes one repo per chip family
(e.g. [`cmsis-device-f4`](https://github.com/STMicroelectronics/cmsis-device-f4))
containing CMSIS device headers, startup files, and example linker configs for every
SKU. Three artifacts bml can consume from there:

| Artifact | Tool | Output |
|---|---|---|
| `Include/<device>.h` `IRQn_Type` + linker MEMORY | [`bml-cmsis`](https://github.com/tralamazza/bml-cmsis) | `.target` file |
| SVD file (NOT in the device repo — see §4) | [`bml-svd`](https://github.com/tralamazza/bml-svd) | `peripheral` declarations |
| `Source/Templates/system_stm32fXxx.c` (optional) | `arm-none-eabi-gcc` + `ld.lld` | `.o` linked alongside bml output |

Flow:

```
cmsis-device-f4 ──bml-cmsis───────> stm32f401xc.target
                \
                 ─system_stm32f4xx.c ──gcc──> system.o ─┐
                                                        │
your_app.bml ───bml build──> app.o, app.ld ─────────────┤── ld.lld ──> app.elf
                                                        │
*.svd ──bml-svd────────────> svd/*.bml ─────────────────┘
                            (imported by your_app.bml)
```

## 2. Generating the target

```
git clone https://github.com/STMicroelectronics/cmsis-device-f4
bml-cmsis --repo ./cmsis-device-f4 --device stm32f401xc --out stm32f401xc.target
```

Where each output field comes from:

| Target field | Source in repo |
|---|---|
| `arch`, `cpu`, `has_bitband` | Hardcoded family table (`f4` -> armv7em + bitband + cortex-m4) |
| `priority_bits`, `has_fpu`, `has_mpu` | `#define __NVIC_PRIO_BITS` / `__FPU_PRESENT` / `__MPU_PRESENT` in `Include/<device>.h` |
| `[mem.flash]`, `[mem.ram]` (`base`/`size`) | `Source/Templates/gcc/linker/*.ld` `MEMORY` block, or (when the repo has no `.ld`) `Source/Templates/iar/linker/*_flash.icf` |
| `vector_table_offset` | Defaults to the flash block's `base` |
| `[interrupts]` | `IRQn_Type` enum in `Include/<device>.h`, NVIC slots only |

Edits you may need to make by hand:

- **`[mem.flash]` size** — the device header covers a whole variant family. For example
  `stm32f103xb.h` is shared by F103C8 (64K) and F103CB (128K); the linker template uses
  the variant maximum. Trim to your actual SKU.
- **`vector_table_offset`** — if you boot under a bootloader (DFU, MCUboot), set this to
  the application offset and adjust the `[mem.flash]` `base`/`size` accordingly.


Refer to a label from bml as `@isr("LABEL", priority=N)`:

```bml
fn tim2_tick() @isr("TIM2", priority=2) {
    // ...
}
```

Slots not listed under `[interrupts]` are filled with `default_handler` by
`bml`'s vector table emitter.

## 4. Peripherals via `bml-svd`

The `cmsis-device-fX` repos do **not** ship SVD files. Get them from one of:

- The STM32Cube MCU firmware package for your family (`STM32CubeF4`, `STM32CubeH7`, ...).
- The community [`cmsis-svd`](https://github.com/cmsis-svd/cmsis-svd-data) mirror.
- ST's [SVD download portal](https://www.st.com).

Then choose one of:

```
# Single combined file containing every peripheral.
bml-svd path/to/stm32f401xc.svd --output src/svd.bml

# One .bml per peripheral, written to the current directory (lowercased
# filenames). cd into the target dir first to control where they land.
mkdir -p src/svd && (cd src/svd && bml-svd ../../path/to/stm32f401xc.svd --split)
```

The `--split` form produces files like `rcc.bml`, `gpioa.bml`, ..., importable
as any other module:

```bml
import svd.rcc;
import svd.gpioa;
```

See [language.md §9](./language.md#9-peripheral-declarations) for the `peripheral`
syntax these files generate.

## 5. C startup interop — calling `SystemInit()`

bml's auto-generated `reset_handler` (defined in `bml-core/src/arch/arm.rs`) sets up
`.data` / `.bss` and jumps to `main`. The vendor `SystemInit()` from
`system_stm32fXxx.c` sits in the same path in a stock STM32 project, doing clock + PLL
setup, FPU enable on M4F, and cache enable on M7. The clean way to integrate it is to
declare it `extern` in bml and call it at the top of `main()`.

### Step 1 — compile the vendor `system_*.c`

`bml cflags --target <file>` prints the `-mcpu` / `-mthumb` / `-mfloat-abi` / `-mfpu`
flags implied by the target file. Substitute the output into your `arm-none-eabi-gcc`
command:

```
arm-none-eabi-gcc -c -O2 \
    $(bml cflags --target stm32f401xc.target) \
    -DSTM32F401xC \
    -I cmsis-device-f4/Include \
    -I CMSIS_5/CMSIS/Core/Include \
    cmsis-device-f4/Source/Templates/system_stm32f4xx.c \
    -o system_stm32f4xx.o
```

For the generated F401xC target this expands to
`-mcpu=cortex-m4 -mthumb -mfloat-abi=hard -mfpu=fpv4-sp-d16`. The `-D<device>` and
`-I...` flags stay project-specific — bml has no opinion on your CMSIS Core include path.
You'll need the [ARM CMSIS Core headers](https://github.com/ARM-software/CMSIS_5) on the
include path because the device header pulls in `core_cm4.h` etc.

`bml cflags` reads `cpu` and `has_fpu` from the target. The default FPU variant for M7
is `fpv5-d16` (double-precision); override on the gcc line if you need
`fpv5-sp-d16` instead. M3 / M0 / M0+ get `-mfloat-abi=soft` and no `-mfpu`.

### Step 2 — declare and call from bml

```bml
extern fn SystemInit();

fn main() @context(thread) {
    SystemInit();
    // ... your init
}
```

No `@context` annotation on the `extern fn` means it's callable from any context (see
[c-interop.md](./c-interop.md)).

**Ordering note.** A stock STM32 reset path runs `SystemInit()` *before* `.data` copy
and `.bss` zero. bml runs `.data` / `.bss` first and then `main`. For every vendor
`SystemInit` shipped by STM (F0 / F1 / F3 / F4 / F7 / L0 / L4 / G0 / G4 / H7), the
function only touches MMIO (RCC, SCB, FPU CP10/CP11) and never reads or writes C
globals, so the order does not matter. If you have a custom `SystemInit` that does
touch C globals, define your own `Reset_Handler` in bml — `bml`'s emitter detects this
and skips its default (see `bml-core/src/arch/arm.rs`).

### Step 3 — link

Pass the C object on the same `ld.lld` line as your bml output. Adapting the
[blue-pill example](../bml/examples/blue-pill/flash.sh):

```
# bml build emits app.o and app.ld
bml build --target stm32f401xc.target your_app.bml

# add system_stm32f4xx.o to the link
ld.lld -T app.ld app.o system_stm32f4xx.o -o app.elf

llvm-objcopy -O binary app.elf app.bin
st-flash write app.bin 0x08000000
```

### Why this isn't automated by `bml-cmsis`

`SystemInit()` lives next to per-chip clock-tree state (`SystemCoreClock`, `HSE_VALUE`,
`PLL_M`/`PLL_N` macros). Most projects edit those defines or override `SystemInit`
entirely with a `SystemClock_Config()`-style function. Vendoring the file into a
generated build also drags in the CMSIS Core include path, which the user is in a
better position to manage. Keeping it a one-line `gcc` rule in the user's build script
is simpler than any automation.

## 6. Limitations and gotchas

- **`[mem.flash]` size** is the variant family maximum, not your SKU. Trim by hand.
- **IRQ label naming** matches the vendor `IRQn_Type` enum, including chip-specific
  composites like `TIM1_BRK_TIM9` (F4) or `DMA1_Channel2_3` (F0). The
  `@isr("LABEL", ...)` string must match exactly.
- **System exceptions** (negative `IRQn` values: NMI, HardFault, SVC, PendSV, SysTick,
  ...) are dropped from the generated `[interrupts]`. bml emits these from a fixed list
  in `bml-core/src/arch/arm.rs` and lets you override them with
  `@isr("NMI", ...)` etc. — see [language.md §4](./language.md#4-interrupt-context-system).
- **No SVD discovery.** SVD files don't live in the `cmsis-device-fX` repos. Get them
  from STM32Cube or the `cmsis-svd` archive and feed them to `bml-svd` separately.
- **No SystemInit auto-import.** See §5 — it's a manual `gcc` + `ld.lld` step.
